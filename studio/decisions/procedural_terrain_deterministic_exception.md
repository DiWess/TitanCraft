# ADR: Deterministic Runtime Terrain Build Is Not "Génération Procédurale Complète"

**Date:** 2026-07-16
**Status:** Accepted
**Owner:** Producer (human decision)
**Reviewed by:** Claude Code (Architecture Validator)

## Context

`README.md` §6 ("Hors périmètre du MVP") forbids *"la génération procédurale complète"* and *"le monde infini"*. `src/World/ProceduralCrashSiteTerrain.cs` (577 lines, instanced live at `scenes/Main/Main.tscn:404`) builds the Crash Site terrain mesh at `_Ready()` from a fixed seed (`ProceduralCrashSiteTerrain.cs:13`), synthesizing grid/zone/height data and facet noise from that seed rather than loading a pre-baked static mesh.

An architecture review flagged this as a literal textual match against the forbidden item: the terrain is generated by code at runtime, which is what "génération procédurale" names. The review did not resolve the question itself and escalated it, per `AGENTS.md` §1 ("if a request conflicts with `README.md`, stop and report the contradiction") and `CLAUDE.md` §6 (scope drift escalates to the Producer).

## Decision

The fixed-seed runtime construction of the single Crash Site terrain mesh is **not** the feature `README.md` §6 forbids, and is accepted as in-scope for the MVP.

## Rationale

- The forbidden items in §6 are `le monde infini` (infinite world) and `la génération procédurale complète` (*complete* procedural generation) — both describing a live system that produces varying or unbounded content across playthroughs (the roguelike/sandbox pattern this MVP explicitly rejects per §5's "une petite carte jouable").
- `ProceduralCrashSiteTerrain.cs` uses a single fixed seed and produces the same one small map every run. It is a code-authored content-construction technique — building geometry from a script instead of a DCC tool — not a generative gameplay system. The output is fixed at author time in the same sense a hand-placed mesh is; only the authoring method differs.
- README §5's "Contenu autorisé" requires exactly "une petite carte jouable" (one small playable map), which this satisfies regardless of whether the mesh was authored in Blender or in C#.
- Treating every runtime-constructed mesh as forbidden would also contradict the already-accepted Blender Asset Forge pipeline, which produces geometry programmatically (via Python/bpy) rather than by hand — the same authoring principle, different tool.

## Consequences

- **Positive:** No rework required; the terrain system stays as-is. The distinction (fixed-seed single-map construction vs. live/varying procedural generation) is now written down so it isn't re-flagged as a contradiction in every future review.
- **Negative:** The line between "acceptable code-authored content" and "forbidden procedural generation" is a judgment call, not a bright line in `README.md` itself. A future feature that reintroduces *variation* between runs (e.g., a random seed, multiple layout variants) would cross back into forbidden territory and is **not** covered by this exception.
- **Neutral:** `README.md` itself is not amended by this ADR (its MVP definition is locked and requires explicit human edit per its own text); this ADR is the supplementary record `studio/decisions/` exists for for exactly this kind of MVP-boundary interpretation call.

## Scope of the Exception

Covered: `src/World/ProceduralCrashSiteTerrain.cs` as it exists today — single fixed seed, single deterministic output, one small map.

Not covered, and would require a new decision: any change that makes the terrain vary across runs/seeds, generates more than one map/zone, or extends this pattern to gameplay-affecting systems (enemy placement, loot tables, mission structure) rather than static dressing/terrain geometry.

## Evidence

- `src/World/ProceduralCrashSiteTerrain.cs:1-577`, `scenes/Main/Main.tscn:404`.

## Verdict

`PASS` — the existing implementation is accepted as in-scope; no code change required as a result of this ADR.
